# StallAlert

A kitesurfing wind-alert system: an Apple Watch app that shows the
next-hour wind forecast and live station measurements for your GPS
position, and fires an audible + haptic alarm when the wind is predicted
to drop below the threshold where your kite stalls — so you can head back
before you're stranded downwind on a dying breeze.

Forecast and station data come from [windguru.cz](https://www.windguru.cz)
(a PRO subscription is required) via a small self-hosted Elixir server.
If the server is unreachable, the watch falls back to fetching a reduced
forecast from Windguru directly.

**This is a personal-use project.** It authenticates to Windguru with your
own PRO account; Windguru offers no official API for this, so treat traffic
politely and keep it personal-scale.

## How it works

```
Apple Watch app  ──HTTPS──▶  Elixir server  ──▶  windguru.cz (PRO account)
      │                        · WG-blend forecast engine (recreates the
      │                          WG model from constituent models)
      │                        · degradation ladder: WG blend → AROME-FR
      │                          1.3 km → GFS 13 km → micro-GFS
      │                        · station readings + nearby candidates
      │                        · caching (15-min forecast TTL, polite
      │                          upstream spacing)
      └──(server down)──▶  micro.windguru.cz directly (GFS only, honest label)
```

- **Server** (`server/`): Elixir, Plug + Bandit, runs as a single Docker
  container behind Caddy (automatic Let's Encrypt TLS). One API endpoint
  the watch polls: `/v1/conditions?lat=..&lon=..[&station_id=..][&model=..]`.
- **Watch app** (`watch/`): SwiftUI watchOS app plus `StallAlertKit`, a
  Swift package holding all testable logic (forecast engine, alert policy,
  compass/trendline/freshness math, clients). Runs an HealthKit workout
  session so alerts keep working with the wrist down.

## Server installation

Prerequisites: a host with a fixed IP, Docker, ports 80/443 open; a DNS A
record for your hostname; a Windguru PRO account.

1. Copy the server to the host and build the image:

   ```bash
   rsync -a --exclude '.git' --exclude '_build' --exclude 'deps' server/ you@host:~/stallalert/server/
   ssh you@host 'cd ~/stallalert/server && docker build -t stallalert-server .'
   ```

2. Create `~/stallalert-deploy.env` on the host (chmod 600), one
   unquoted `KEY=value` per line — do **not** `source` it:

   ```
   API_TOKEN=<random secret; the watch app needs the same one>
   WG_USERNAME=<windguru login>
   WG_PASSWORD=<windguru password; reserved, not used yet>
   WG_MICRO_PASSWORD=<micro.windguru.cz API password>
   WG_COOKIE=<FULL Cookie header from a logged-in windguru.cz browser session>
   WG_SPOT_ID=<windguru spot id near home; drives the WG-blend weights>
   ```

   Only `API_TOKEN` is required to boot. `WG_COOKIE` gates the primary
   forecast source and cannot be obtained programmatically — when it
   expires (the served `forecast.model` becomes `"gfs-micro"`), log into
   windguru.cz in a browser, copy the full Cookie request-header value,
   update the env file, and restart the container.

3. Run it, bound to localhost only:

   ```bash
   docker run -d --restart unless-stopped --name stallalert \
     -p 127.0.0.1:4000:4000 --env-file ~/stallalert-deploy.env stallalert-server
   ```

4. Put Caddy in front for TLS — append to your Caddyfile and reload:

   ```
   your.domain.tld {
       reverse_proxy 127.0.0.1:4000
   }
   ```

5. Verify:

   ```bash
   curl https://your.domain.tld/v1/health
   # {"status":"ok"}
   curl -H "Authorization: Bearer $API_TOKEN" \
     "https://your.domain.tld/v1/conditions?lat=39.92&lon=3.09"
   # 200 with forecast (model "WG blend (N models)" once warm), station,
   # nearby_stations, available_models
   ```

Day-to-day redeploys: `./redeploy-server.sh` on the host (copy it up once
with scp). The full deployment guide, including host-specific quirks and
smoke tests, is in [docs/deploy.md](docs/deploy.md).

## Watch app installation

Built with Xcode from `watch/` (project generated by XcodeGen from
`project.yml`; regenerating **wipes the scheme's credential values** — see
below). Deployment target watchOS 10; developed against an Apple Watch
Ultra 2.

1. Open `watch/StallAlert.xcodeproj`, set your signing team, select your
   watch as the destination (VPNs off and the iPhone cabled make the
   Xcode-to-watch tunnel far more reliable), and Run.
2. Credentials (DEBUG builds only) seed from scheme environment variables
   on first launch — edit the StallAlert scheme's Run environment:
   `STALLALERT_SERVICE_URL`, `STALLALERT_SERVICE_TOKEN` (the server's
   `API_TOKEN`), `STALLALERT_WG_USERNAME`, `STALLALERT_WG_MICRO_PASSWORD`.
   They can also be entered on the watch in Settings.
3. Verify against [docs/hardware-checklist.md](docs/hardware-checklist.md).

## Using the app

**Start screen** — shows the next-hour forecast for your position once
data arrives, the alert threshold, and the Start Session button (a
"Starting…" spinner covers the brief HealthKit spin-up). Settings lets you
set the threshold in knots, the service URL/credentials, and the forecast
source (WG blend by default, or a specific model).

**Session screen**, top to bottom:

- **NEXT HOUR** — predicted range in knots (colored green/orange/red by
  distance to your threshold), a mini trendline of the coming hour's base
  wind against a dashed red threshold line, and a dial showing the
  forecast wind direction (downwind convention, like Windguru's arrows).
  A small caption appears only when the served model differs from the one
  you requested (degradation is visible, normality is silent). A
  "dropping to ~N" line warns of a falling trend.
- **NOW** — the live station: name and distance (tap to pick a different
  nearby station — a pin marks a manual choice, which sticks for that spot
  until you reset to auto), wind and gust, and a compass whose arrow points
  downwind with fading "shadow" ticks tracing the last hour of direction
  history. Tap the numbers to force a refresh (haptic click, line dims
  while fetching).
- **Freshness line** — "measured n min ago" flashes greyish-green on a new
  station sample and fades to gray in 5 minutes. On the dotted track, the
  **green** chevron resets on every successful fetch and the **blue** one
  ages with the station sample; both travel right over 15 minutes (blue
  becomes a clock when the sample is old). Green resetting while blue
  drifts = your connection is fine, the station is just slow; both
  drifting together = you're offline.
- **Status row** — the alert threshold, plus an orange `direct` pill when
  the watch has failed over to fetching from Windguru directly (reduced,
  GFS-only forecast) and a red `offline` pill when nothing is reachable.

**Alerts** fire once per event — when the forecast predicts the base wind
dropping below your threshold within the hour, or when the live reading
falls below it — as three rounds of strong haptics plus an alarm tone
(~21 s; watch Silent Mode mutes the tone, never the haptics). Tap OK to
acknowledge. Alerts keep working offline from cached data until it ages
out.

**Water lock is not enabled automatically** — flick the water-drop icon in
Control Center when you actually get on the water; otherwise it would
block tapping OK on alerts (water lock ignores all touches until a
crown-turn eject).
